CyberScripts uses a two-phase deployment model to push and activate the security suite across every host in your competition range.

## Two-phase deployment

**Phase 1 — Push (`deploy.sh`):** Run once from your control machine. It compresses all scripts into a tarball, then uses `sshpass` and GNU `parallel` to SCP the files to every host and immediately run the firewall lockdown and hardening scripts — all in parallel, up to 10 hosts at a time.

**Phase 2 — Activate (`activate.sh`):** Run on each target host after `deploy.sh` completes. It unpacks the tarball, installs auditd, deploys Watchdawg, installs BusyBox, and hardens the SSH configuration.

## What gets deployed

`deploy.sh` packages the following files into `minzero.tar.gz` before copying them to each host:

| File | Purpose |
|---|---|
| `c2scanner.sh` | C2 connection monitor |
| `sshd_config` | Hardened SSH server configuration |
| `backup.sh` | System state snapshot at competition start |
| `watchdawg.sh` | File integrity monitor |
| `watchdawg-sources` | Source list for Watchdawg to monitor |
| `auditd-rules` | auditd ruleset for the competition |

In addition to the tarball, `deploy.sh` uploads these files directly to `/tmp/` on each host:

- `harden.sh` — user and shell hardening
- `autofirewall.sh` — iptables lockdown
- `activate.sh` — phase 2 activation script
- `pass` — temporary sudo password helper (shredded after use)
- `port-sources` — host-specific inbound/outbound port list

## The `systems/` directory

The `systems/` directory on your control machine contains one subdirectory per host. Each subdirectory name must exactly match the second column (`<system-name>`) of the hostfile.

```
systems/
├── webserver/
│   └── port-sources
├── database/
│   └── port-sources
└── dns/
    └── port-sources
```

The `port-sources` file controls which TCP ports `autofirewall.sh` opens on that host:

- **Line 1:** space-separated inbound TCP ports
- **Line 2:** space-separated outbound TCP ports (port 53 also gets a UDP rule)

## The hostfile

The hostfile lives in `minzero/` and has one entry per host:

```
<ip> <system-name> <password-hash>
```

A real example:

```
192.168.1.10 webserver $6$a3f8c2d1e4b7f9a0$Kx2LvExampleHashAbcDef1234567890abcdef
192.168.1.20 database $6$a3f8c2d1e4b7f9a0$Kx2LvExampleHashAbcDef1234567890abcdef
192.168.1.30 dns $6$b9e1d4f2c7a8b3e5$Pq7MwExampleHashXyzUvw0987654321fedcba
```

| Field | Description |
|---|---|
| `<ip>` | Target host IP address |
| `<system-name>` | Short identifier; must match a directory under `systems/` |
| `<password-hash>` | SHA-512 hash generated by `funcs/passwordHasher.sh` |

## Parallel execution

`deploy.sh` uses GNU `parallel -j 10` to process up to 10 hosts simultaneously. Each host runs an independent `deploy_host()` function that handles its own SCP transfer and remote command execution. All output — from every host — is streamed to the terminal with a `[system-name]` prefix and appended to `deploy.log`.

```
[webserver] Begin system 192.168.1.10 with user root
[database]  Begin system 192.168.1.20 with user root
[webserver] starting autofirewall.sh
[database]  starting autofirewall.sh
[webserver] done autofirewall.sh
[webserver] starting harden.sh
...
```

## Deployment log

All output from `deploy.sh` — including per-host progress, remote command output, and errors — is appended to `minzero/deploy.log` via `tee`. Review it after a run to confirm every host completed successfully.

```bash
tail -f minzero/deploy.log
```

<Note>
If a host fails mid-run (SCP error, SSH timeout, sudo failure), its `[system-name]` prefix makes it easy to spot in the log. Re-run `deploy.sh` targeting the specific host by temporarily reducing the hostfile to that entry.
</Note>

title: 'SSH configuration'
description: 'Hardened sshd_config deployed to all competition hosts'

CyberScripts ships a hardened `sshd_config` that is deployed to every competition host by `activate.sh`. The configuration trades convenience features for a minimal attack surface: public-key authentication is disabled in favour of password auth (using the `bluey` account), root login is forbidden, and idle sessions are terminated automatically.

## Full configuration

```text
PermitRootLogin no # this one is really important!
PasswordAuthentication yes
IgnoreUserKnownHosts yes
IgnoreRhosts yes
PermitEmptyPasswords no
PubkeyAuthentication no        # ssh keys
IgnoreRhosts yes
UsePAM yes
X11Forwarding no
UsePrivilegeSeparation sandbox # (if that breaks, change sandbox to yes)
RSAAuthentication no

ClientAliveInterval 300
ClientAliveCountMax 0
```

## Directive reference

### PermitRootLogin no

Prevents direct root login over SSH. Attackers who obtain SSH credentials can never log in as root directly — they must first authenticate as `bluey` and then escalate via `sudo`. This is one of the most impactful settings in the file.

### PasswordAuthentication yes

Keeps password-based authentication enabled. In competition environments, distributing SSH key pairs to all team members introduces operational overhead and risk of key loss. The `bluey` account is protected by a strong pre-generated hash set by `harden.sh`, so password auth remains the team's login method.

### PubkeyAuthentication no

Explicitly disables public-key authentication. This prevents an attacker who has planted an `authorized_keys` file from maintaining persistence through SSH keys, even if they had brief write access to a home directory.

### PermitEmptyPasswords no

Refuses authentication attempts where the password field is empty. This is a hard safeguard against misconfigured accounts or edge cases in PAM stacks that might otherwise permit a blank password.

### IgnoreUserKnownHosts yes

Tells `sshd` to ignore each user's `~/.ssh/known_hosts` file when evaluating host-based authentication. Combined with `IgnoreRhosts yes`, this fully disables the legacy host-based authentication mechanisms.

### IgnoreRhosts yes

Disables `.rhosts` and `.shosts` files for host-based authentication. These legacy mechanisms are trivially spoofable and should never be active in a competition environment. This directive appears twice in the file; the second entry is redundant but harmless.

### UsePAM yes

Routes authentication through PAM (Pluggable Authentication Modules). Required on most modern distributions for account and session management (e.g., `pam_unix`, `pam_limits`, `pam_loginuid`). Disabling PAM can break `su`, `sudo`, and audit logging on some systems.

### X11Forwarding no

Disables X11 display forwarding. There are no graphical applications in a competition shell environment, and X11 forwarding has a history of vulnerabilities. Turning it off removes both the attack surface and unnecessary network overhead.

### UsePrivilegeSeparation sandbox

Runs the unprivileged SSH pre-authentication child process in a restricted sandbox (using OS-level sandboxing such as `pledge` on OpenBSD or `seccomp` on Linux). If sandboxing is unavailable or causes issues on the target host, the comment in the file notes it can be changed to `yes` (non-sandboxed privilege separation) as a fallback.

### RSAAuthentication no

Disables the legacy SSHv1 `RSAAuthentication` mechanism. This directive is a no-op on modern `sshd` versions that only support SSHv2, but it is included explicitly to prevent any ambiguity.

### ClientAliveInterval 300

Sends a keepalive probe to the client every 300 seconds (5 minutes) of inactivity. If the client does not respond within the `ClientAliveCountMax` window, the session is terminated.

### ClientAliveCountMax 0

Sets the number of unanswered keepalive probes before disconnecting to `0`. Combined with `ClientAliveInterval 300`, this means the session is dropped **immediately** on the first missed keepalive — effectively terminating idle sessions after exactly 5 minutes of inactivity. This limits the window an attacker has if a team member walks away from an open terminal.

## Deployment

`activate.sh` handles deploying this config to a host:

1. Backs up the existing config to `/etc/ssh/.old`
2. Copies the new `sshd_config` into place at `/etc/ssh/sshd_config`
3. Restarts the SSH daemon to apply the new settings
4. Runs `chattr +i /etc/ssh/sshd_config` to make the file immutable

<Warning>
  SSH configuration changes take effect only **after the daemon restarts**. If you need to modify the deployed config (e.g., to add an `AllowUsers` directive), you must first remove the immutable flag:

  ```bash
  chattr -i /etc/ssh/sshd_config
  ```

  Edit the file, then restart SSH and re-apply the immutable flag. If your changes contain a syntax error, `sshd` will refuse to restart — **test with `sshd -t` first** to avoid locking yourself out.
</Warning>

<Tip>
  To validate the active configuration without restarting `sshd`:

  ```bash
  sshd -T | grep -E 'permitrootlogin|passwordauthentication|pubkeyauthentication|x11forwarding'
  ```
</Tip>
